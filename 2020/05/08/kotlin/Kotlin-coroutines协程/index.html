结构化并发

优点

  • CoroutineScope 会跟踪所有协程，并且可以取消由它所启动的所有协程。
  • 结构化并发能够保证当某个作用域被取消后，它内部所创建的所有协程也都被取消。
  • 结构化并发保证了当 suspend 函数返回时，它所处理任务也都已完成。
  • coroutineScope 和 supervisorScope 将会等待所有的子协程都完成。
  • 结构化并发保证当一个协程出错时，它的调用方或作用域会被通知到。

适用性

协程在以下两种场景中非常好用，

  • 一次性请求 (one shot requests) 是那种调用一下就请求一下，请求获取到结果后就结束执行；

  • 流式请求 (streaming request) 在发出请求后，还一直监听它的变化并返回给调用方，在拿到第一个结果之后它们也不会结束。

并发问题

kotlin 协程的并发问题，有三个基本的模式可以让我们确保在同一时间只会有一次请求进行:

  1. 在启动更多协程之前取消之前的任务；

  2. 让下一个任务排队等待前一个任务执行完成；

  3. 如果有一个任务正在执行，返回该任务，而不是启动一个新的任务。

Best Practices

在实际开发中，我们更多的会采用结构化并发协程，即定义协程时指定 CoroutineScope 。CoroutineScope 会跟踪所有协程，同样它还可以取消由它所启动的所有协程。这在 Android 开发中非常有用，比如它能够在用户离开界面时停止执行协程。

协程启动方式有两种，

  • launch 构建器适合执行 “一劳永逸” 的工作，意思就是说它可以启动新协程而不将结果返回给调用方
  • asyn 构建器可启动新协程并允许您使用一个名为 await 的挂起函数返回 result

注意：Kotlin 禁止您创建不能够被追踪的协程，从而避免协程泄漏

delay

最简单的协程， 通过 Activity 中 lifecycleScope 启动一个协程，delay(xx) 模拟延迟操作，实现常见的 Splash 页面延迟跳转。
Fragment 中也有内置的 lifecycleScope 对象可以使用。

override fun onViewsReady() {
    // launch 启动协程
    lifecycleScope.launchWhenResumed { navigateTo() }
}

private suspend fun navigateTo() {
    delay(3000L)
    Intent(this@SplashActivity, LoginActivity::class.java)
        .run {
            startActivity(this)
            this@SplashActivity.finish()
        }
}
